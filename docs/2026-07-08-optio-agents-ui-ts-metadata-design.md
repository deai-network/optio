# optio-agents-ui — canonical agent metadata for TS consumers

**Date:** 2026-07-08
**Status:** Design approved, pending implementation plan
**Follows:** [2026-07-08-agent-canonical-metadata-design.md](2026-07-08-agent-canonical-metadata-design.md) (the Python side)

## Problem

The Python side now exposes canonical agent metadata (`AgentInfo{slug,name,url}`
per engine, aggregated as `optio_agents_all.AGENTS` + `get_agent_info()`). TS
frontends and external TS consumers have no equivalent: the canonical **names**
and **URLs** live only in Python, and the 7-engine **slug set** is hand-duplicated
in ≥3 places on the TS side (`SpinnerEngine` union in
`optio-conversation-ui/src/spinners/NativeSpinner.tsx`, the `ConversationWidget`
literal chain, per-View `engine="…"` literals).

## Goal

Give TS consumers — the in-repo frontends **and external/third-party TS apps** —
the same canonical agent metadata, from a single source of truth, with no manual
cross-language duplication.

## Constraints

- **Agents are external to core.** `optio-contracts` and the core packages
  (`optio-core`, `optio-api`, `optio-ui`, `filtrum-*`) cover the core optio
  protocol only; agents are just tasks like any other. Core must **not** depend on
  the agent layer. So the metadata cannot live in `optio-contracts`.
- **Both artifacts are published.** The Python `optio-agents-all` and the new TS
  package each ship independently, so each must **embed** its own copy of the data
  (a shared repo-root file read at runtime would not be bundled into either dist).
  Therefore one side is authoritative and the other is **generated from it** at
  build time and committed.
- **Python stays the single source of truth.** The per-engine `info.py`
  (`AGENT_INFO`) constants + `optio_agents_all.AGENTS` shipped in the prior work are
  authoritative. TS is derived.

## Architecture

A new published TS package `optio-agents-ui` — the TS home of the agent layer,
sibling in spirit to Python's `optio_agents_all`. Its metadata is **generated from
Python's `AGENTS`** by a small emitter hooked into `make codegen`; the output is
committed (ships in the tarball, no Python needed for normal `tsc`/publish).

### A. Package `optio-agents-ui`

```
packages/optio-agents-ui/
  src/
    agents.generated.ts   # GENERATED + committed: AgentType union + AGENTS data
    agent-info.ts         # hand-written: AgentInfo interface + getAgentInfo()
    index.ts              # re-exports AgentType, AgentInfo, AGENTS, getAgentInfo
  scripts/
    generate.py           # the Python->TS emitter (see B)
  package.json            # tsc build, ESM, files: ["dist"], publishable (unscoped)
  tsconfig.json           # extends tsconfig.base.json
```

Public API (mirrors `optio_agents_all`):

```ts
export type AgentType = 'antigravity' | 'claudecode' | 'codex' | 'cursor'
  | 'grok' | 'kimicode' | 'opencode';               // generated (sorted)
export interface AgentInfo { slug: AgentType; name: string; url: string; }
export const AGENTS: Record<AgentType, AgentInfo>;   // generated
export function getAgentInfo(slug: AgentType): AgentInfo;   // AGENTS[slug]
```

- External to core: **no** dependency on `optio-contracts`/core packages.
- `AgentInfo` interface and `getAgentInfo` are hand-written in `agent-info.ts`
  (stable, not data); only the union + data table are generated.

### B. Generation (Python → TS)

`packages/optio-agents-ui/scripts/generate.py`:

1. `from optio_agents_all import AGENTS`.
2. Emit `src/agents.generated.ts`:
   - a `// AUTO-GENERATED by scripts/generate.py from optio_agents_all.AGENTS. DO NOT EDIT.`
     header (matching the existing `_generated` convention),
   - the `AgentType` union built from `sorted(AGENTS.keys())`,
   - the `AGENTS: Record<AgentType, AgentInfo>` const literal, keys in sorted order,
     each `{ slug, name, url }`.

Wiring:

- Hook the emitter into the existing `make codegen` target (runs with
  `clamator-codegen`), invoked via the repo `.venv` Python.
- Output is **committed** — survives `make clean`, removed only by `clean-codegen`
  (same treatment as the existing `_generated/` dirs).
- **Determinism:** sorted keys + stable formatting → byte-identical on re-run. A CI
  guard asserts "no diff after regenerate" (codegen-clean check).

### C. Consumers (dedup the slug set — scope 1 + 3)

- `optio-conversation-ui` adds a workspace dependency on `optio-agents-ui`.
- `SpinnerEngine` union → replaced by imported `AgentType`;
  `BUILDERS: Record<AgentType, …>` retains exhaustiveness.
- `ConversationWidget.tsx` engine dispatch + per-View `engine="…"` literals are
  typed against `AgentType`, sourcing the slug set from the shared package instead
  of re-declaring it.
- **No new on-screen text.** Rendering canonical names in the UI is intentionally
  out of scope (a later, separate UI decision).

### D. Build & release wiring

- `pnpm-workspace.yaml` already globs `packages/*` — no change needed.
- Add `optio-agents-ui` to `RELEASABLE_TS` (Makefile) and `TS_PUBLISHABLE`
  (`scripts/release/run.py`).
- Release dependency order: `optio-agents-ui` **before** `optio-conversation-ui`.

## Testing

- **Generation:** running the emitter yields byte-stable `agents.generated.ts`
  (codegen-clean guard); a snapshot/equality test asserts the generated `AGENTS`
  covers exactly the 7 engines with the canonical name/url (the generator is the
  guarantee; the test catches accidental drift or a broken emitter).
- **Consumers:** `tsc` typechecks `optio-agents-ui` and `optio-conversation-ui`;
  existing conversation-ui tests stay green.
- Full `make test` (TS + Python) green.

## Out of scope

- Rendering canonical agent names/URLs in any UI surface (separate later call).
- The demo-prose canonicalization noted after the Python work.
- Any change to core packages or `optio-contracts`.
- Runtime agent-catalog HTTP endpoint (not needed for static display metadata).
